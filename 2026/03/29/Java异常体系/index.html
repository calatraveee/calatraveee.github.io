Java异常体系

发表于2026-03-29|更新于2026-03-29|基础知识
|浏览量:

整体架构

Java异常体系以Throwable为根类，分为两大分支：Error和Exception。Error为严重错误，是程序无法处理的。Exception是程序可处理的异常，分为：IOException、SQLException、ClassNotFoundException和RuntimeException。其中IOException为检查异常（check），RuntimeException为非检查异常（Unchecked）。

3种类型详解

Error（错误）

本质：JVM层面的严重问题，程序员无法也不应该捕获处理，属于Unchecked，编译器不强制处理。处理原则：不要catch，让程序崩溃，靠运维/监控介入

Checked Exception（检查异常）

本质：编译器强制要求处理的异常，必须try-catch或throws声明。直接继承Exception（非RuntimeException），调用方可以合理预期并有能力恢复的异常。

Unchecked Exception（非检查异常/运行时异常）

本质，继承自RuntimeException，编译器不强制处理。通常是编程错误，如空指针、越界，应该修代码而非捕获。

维度 Checked Exception Unchecked Exception
继承关系 Exception 的直接子类 RuntimeException 的子类
编译器检查 ✅ 强制处理 ❌ 无需处理
设计语义 外部因素，可恢复 编程错误，应修复代码
使用场景 IO、网络、数据库 参数校验、逻辑错误
API 文档 必须在方法签名中声明 可选声明

设计规范

何时用Checked Exception：

  1. 调用方有能力且应该处理这个异常
  2. 异常来自外部系统（文件、网络、数据库）
  3. 失败是合理预期的，而非编程错误

何时用Checked Exception：

  1. 调用方传入了非法参数（前置条件违反）
  2. 编程逻辑错误，修代码才是正道
  3. 异常在绝大多数调用栈中无法被合理处理

现代最佳实践总结

  1. 优先使用 Unchecked Exception：Spring、Hibernate 等主流框架都已转向 Unchecked，减少模板代码
  2. Checked Exception 用于边界层：只在 IO/网络等真正需要调用方感知的地方使用
  3. 分层包装异常：DAO 层 → Service 层 → Controller 层，每层翻译成对应语义
  4. 永远不要吞异常：至少要 log.error，绝不静默忽略
  5. 异常信息要有诊断价值：包含上下文参数，方便排查

Check异常是“合同”–你调用我，你必须处理可能出现的意外；Unchecked异常是“警告”–这是代码的问题，应该修代码。

初识单调栈：碗子数组的数目
碗子数组的数目给你一个整数数组 nums，包含 互不相同 的元素。nums 的一个子数组 nums[l…r] 被称为 碗（bowl），如果它满足以下条件： 子数组的长度至少为 3。也就是说，r - l + 1 >= 3。其两端元素的 最小值 严格大于 中间所有元素的 最大值。也就是说，min(nums[l], nums[r]) > max(nums[l + 1], …, nums[r - 1])。返回 nums 中 碗 子数组的数量。 子数组 是数组中连续的元素序列。 示例1: 输入: nums = [2,5,3,1,4]输出: 2解释:碗子数组是 [3, 1, 4] 和 [5, 3, 1, 4]。[3, 1, 4] 是一个碗，因为 min(3, 4) = 3 > max(1) = 1。[5, 3, 1, 4] 是一个碗，因为 min(5, 4) = 4 > max(3, 1) = 3。
